# Финальный план: умные родительские фильтры и живой индикатор статуса
### Pattaya Kids Guide — синтез после четырёх критик

Вероника, ниже — не пересказ черновика, а решение. Четыре критика (UX, Next-инженер, родитель-удалёнщик, data-архитектор) во многом сошлись, и там, где сошлись — я это принял. Там, где спорили — я выбрал позицию и объясняю почему. Несколько «фактов» из критик я перепроверил в коде: часть оказалась неверной, и это поменяло выводы (об этом честно в тексте).

**Три сквозных вывода, на которых держится весь план (все четыре критика пришли к ним независимо):**

1. **Фичу нельзя привязывать к слову «сад».** У всех садов разное время (тайский — до 14:30, международный — до 17:00, мини-садики — до 13:00). Привязка к «саду» встраивает *ваш личный* график в логику всего продукта и заставляет фильтр врать. Фича — про **потребность работающего родителя**, а не про время сада.
2. **Разделить два вопроса родителя:** «куда вообще можно» (это фильтр = свойство места, статика, кешируется) и «успею ли сегодня» (это индикатор в карточке = момент времени, динамика). Не смешивать их в одном фильтре — тогда исчезает вся сложность («статический vs живой режим», денормализация, hydration).
3. **Не переусложнять модель.** Денормализованный флаг `worksAfterDaycareCached` — выкинуть (создаёт тихие баги). Enum языков — заменить на справочник. Никаких новых индексов (нужный уже есть в коде).

## Раздел 1. Новые признаки места: что и как моделируем

### Правило «булево vs справочник vs отдельная модель» (зафиксировать в схеме)

Это правило разграничения нужно **записать комментарием прямо в `schema.prisma` над моделью `Place`** — иначе следующий признак снова уедет «куда попало» (совет data-архитектора, принято):

| Механизм | Когда | Стоимость |
|---|---|---|
| **Булево на `Place`** | признак участвует в фильтре как жёсткое да/нет, есть почти везде, влияет на логику фичи | дешёвый `where`, ноль join |
| **Справочник (M:N, скоуп страны)** | «одно из многих» в открытом растущем списке, набор значений зависит от страны | гибкость без миграций |
| **Отдельная модель 1:1** | у признака внутренняя структура (несколько полей + уточнения) | как `PlaceBirthdayInfo` |

### 1.1. Кондиционер — булево `hasAirCon` ✅ (все четыре критика согласны)

`indoor ≠ AC` (крытая открытая зона — тоже `indoor`, но без кондиционера). Это условие бизнес-логики «можно поработать», в справочник его нельзя (пришлось бы join на каждый запрос). AC актуален в любой жаркой стране → булево масштабируется.

```prisma
hasAirCon Boolean @default(false)  // отдельно от indoor: крытая зона ≠ кондиционер
```

### 1.2. Парковка — булево `hasParking` ✅ (все согласны)

Поднимаем с уровня `Amenity` до первоклассного фильтра, как вы просили.

```prisma
hasParking Boolean @default(false)
```

**Важная поправка от data-архитектора (принято, экономит вам работу):** черновик предлагал написать «скрипт миграции данных» из `Amenity(parking)` в колонку. **Это не нужно.** У вас нет Prisma migrations — база пересоздаётся через `db push` + `tsx` seed. Значит: если `Amenity(parking)` есть в сиде — просто уберите его из сида и добавьте `hasParking` местам там же. Одно место истины — колонка. Никаких скриптов.

Нюансы (платная/бесплатная/скутеры) — **не в `description`** (там их не выудить фильтром), а позже отдельным `enum ParkingType` или полем `parkingNote`. Не сейчас.

### 1.3. «Можно поработать» — два новых честных булева вместо переиспользования `hasFood`

Здесь **все три содержательных критика (UX, инженер, родитель) независимо забраковали `hasFood` как прокси «кафе»**. `hasFood = true` стоит у фудкорта, вендинга, ларька с мороженым — туда с ноутбуком на 2 часа не сядешь. Переиспользовать `hasFood` — значит наполнить киллер-фичу ложными срабатываниями, а это бьёт ровно по доверию, которое для вас важнее отзывов.

**Решение:** завести честный булев под саму суть фичи + добавить признак, который родитель-удалёнщик назвал «важнее парковки»:

```prisma
hasCafeSeating   Boolean @default(false)  // есть где взрослому сесть с кофе надолго (не просто «есть еда»)
hasPowerOutlets  Boolean @default(false)  // розетки — для работы критичнее парковки: ноут сядет раньше ребёнка
```

`hasFood` остаётся как есть — общий тег «есть перекусить», но **из условия фичи уходит**.

> **Trade-off:** два новых поля вместо нуля — но они закрывают *суть* киллер-фичи, а не приближают её. Одно честное булево дешевле, чем семантический хак, который тихо подрывает доверие. Родитель-удалёнщик: «розетка важнее парковки», а её в черновике не было вовсе.

### 1.4. Язык персонала — справочник `Language` (ISO-код), НЕ Prisma enum

Черновик предлагал `enum StaffLanguage { EN TH RU ZH }`. **Data-архитектор поймал здесь противоречие вашему же зафиксированному принципу** («поле есть не во всех странах → справочник»): Prisma-enum требует **миграцию схемы на каждую новую страну** (VN, KH, DE, AR…). Это ровно та болезнь мультирегиона, от которой вы хотели уйти.

```prisma
model Language {
  code  String @id   // ISO 639-1: "en","th","ru","zh"
  name  String
}

model PlaceStaffLanguage {
  placeId      String
  languageCode String
  place        Place    @relation(fields: [placeId], references: [id])
  language     Language @relation(fields: [languageCode], references: [code])
  @@id([placeId, languageCode])
  @@index([languageCode])
}
```

Новый язык = вставка одной строки, не миграция. ISO-код бесплатно стыкуется с `[lang]` в роутинге. Типобезопасность enum здесь ложная цель — языки это **данные страны**, а не константы кода.

> **Позиция по MVP:** модель заложить сейчас (на пустой базе дёшево, принцип «модель широкая сразу»), показывать язык **тегом** на карточке — можно. А вот **фильтр-чекбокс «англоговорящий персонал» — отложить**: пока данные о языке нигде не собраны, фильтр даст пустоту (тот же аргумент, по которому отложены стульчики). Тег — да, фильтр — рано.

### 1.5. Детское кафе — отдельная модель `PlaceKidsCafe` (1:1), отложить; но стульчики решить сознательно

Паттерн 1:1 в проекте есть (`PlaceBirthdayInfo`). «Детское кафе» — это три разные вещи (меню, здоровая готовка, стульчики), не одно булево.

**Развилка, которую подсветил data-архитектор:** черновик сам назвал детские стульчики «самым частым вопросом родителя», но хоронит их внутри отложенной nullable-модели, по которой фильтровать дорого и поздно. Нельзя одновременно говорить «киллер-вопрос» и «отложим фильтруемость».

**Моё решение:** стульчики — это **не киллер-вопрос уровня AC/розетки** (родитель-удалёнщик их даже не упомянул, ему важнее «можно ли оставить ребёнка» и «тихо ли»). Поэтому стульчики остаются внутри `PlaceKidsCafe` и **вся модель отложена в post-MVP**. Если наполнение покажет, что стульчики спрашивают чаще прочего — поднимем `hasHighChairs` до булева на `Place` (это дешёвая правка потом). Сейчас — не тащим.

```prisma
// POST-MVP
model PlaceKidsCafe {
  id             String  @id @default(cuid())
  placeId        String  @unique
  hasKidsMenu    Boolean @default(false)
  hasHighChairs  Boolean @default(false)
  healthyCooking Boolean @default(false)  // напр. аэрогриль, без масла
  notes          String?
  place          Place   @relation(fields: [placeId], references: [id])
}
```

### 1.6. Индексы — НЕ добавляем (перепроверено в коде: нужный уже есть)

Data-архитектор утверждал, что композитного `@@index([cityId, status])` нет и его надо добавить. **Я проверил — он уже есть** (`schema.prisma:201`). А индексы по отдельным булевам на объёме «десятки мест на город» бесполезны: селективность булева низкая, `cityId + status` уже сужают выборку до единиц строк, планировщик возьмёт seq scan. **Итог: ноль новых индексов.** Не притворяемся, что «булево = скорость» на этом масштабе.

### 1.7. Мультиинтервальные дни (обеденные перерывы) — схему не трогаем, чиним UI

`PlaceSchedule` не имеет `@@unique([placeId, day])` → несколько строк на день (09:00–14:00 + 17:00–21:00) **уже физически возможны**, менять схему не нужно. Но два дефекта:

- **Баг:** `key={schedule.day}` (детали, строка 101) на дубле дня даст React-варнинг → чинить на `key={`${schedule.day}-${index}`}`.
- **Недетерминизм:** без `@@unique` порядок строк дня не гарантирован → добавить `orderBy: [{ day: "asc" }, { openTime: "asc" }]` при выборке `schedules`, иначе интервалы «прыгают».
- **Страховка (совет UX):** добавить `@@unique([placeId, day, openTime])` — разрешает мультиинтервалы, но блокирует полные дубли при кривом импорте. Дёшево.

```prisma
model PlaceSchedule {
  // ...без изменений полей...
  @@unique([placeId, day, openTime])  // разрешить обед (2 интервала), запретить полный дубль
}
```

## Раздел 2. Фильтр «Можно поработать» (бывший «После сада»): финальное определение

### Переименование — принципиальное, не косметика

Все критики сошлись: фильтр называть **«Можно поработать»** (варианты копи: «С ребёнком и ноутбуком», «Поработать, пока ребёнок занят»). Слово «сад» уходит из названия. Причина: фича описывает удобства **для родителя** (wifi+AC+кафе+розетка), а не потребность ребёнка и не время конкретного сада. Если оставить название «После сада» — оно врёт про время и отсекает половину аудитории (родителей, кому сад не в 15:00).

### Финальное определение (чистая статика, без расписания в фильтре)

Место проходит фильтр «Можно поработать», если **одновременно**:

```
hasWifi = true  И  hasAirCon = true  И  hasCafeSeating = true
```

**Это всё.** Никакого расписания, никакого окна времени, никакой денормализации в самом фильтре.

> **Это моё главное расхождение с черновиком, и я объясню развёрнуто.** Черновик встроил в фильтр условие «работает вечером в будни» + окно 15:00–20:00. **Все четыре критика это забраковали, по трём разным причинам:**
> - *Родитель:* окно 15:00–20:00 привязано к вашему саду; место 10:00–15:00 формально «пересекает 15:00» и попадёт в выдачу, хотя приедешь в 15:40 — закрыто. Фильтр врёт в первый день.
> - *Data-архитектор + Next-инженер:* время в фильтре тянет за собой денормализованный флаг `worksAfterDaycareCached`, а он создаёт класс тихих багов «поменял расписание → забыл пересчитать → фильтр молча врёт». Для соло-фаундера 3 ч/день без тестов это худший размен.
> - *UX:* «успею ли сегодня» — это не свойство места (фильтр), это момент времени (индикатор). Смешивать их в фильтре — концептуальная ошибка.
>
> **Вопрос «успею ли доехать сегодня» полностью переезжает в живой индикатор карточки** (Раздел 3). Родитель применяет статический фильтр «Можно поработать» → видит места с нужными удобствами → и **глазами по бейджам** видит, где открыто ещё надолго. Ваше решение №2 («успею доехать» ≈ «открыто ещё ≥2 ч») этим **сохраняется полностью** — просто живёт в бейдже, а не в SQL. Развилка «статический vs живой режим» из черновика **исчезает**: она была artefact’ом от смешивания двух вопросов.

### Что с `worksAfterDaycareCached`? — Удалить из плана целиком

Денормализованный флаг не нужен, потому что расписание вообще ушло из фильтра. Если позже захотите добавить «и работает вечером» как *опцию* — это делается relation-`where` **без денормализации** (лексикографическое сравнение `"HH:MM"` в Postgres корректно для zero-padded формата, я это перепроверил — два инженерных критика подтверждают):

```ts
// ОПЦИЯ на будущее, НЕ в MVP — если понадобится «работает вечером в будни»:
schedules: { some: {
  isClosed: false,
  day: { in: ["MON","TUE","WED","THU","FRI"] },
  openTime:  { lt: "20:00" },   // строковое сравнение корректно для "HH:MM"
  closeTime: { gt: "15:00" },
}}
```

На объёме «десятки мест» это не ломает пагинацию и не тормозит. Но **в MVP — не добавляем даже это.** Фильтр = три булева.

### Технически: `canWork` — отдельная ветка в `buildApprovedPlacesWhere`

Перепроверено: `buildApprovedPlacesWhere` — плоский, прокидывает булевы один-в-один. `canWork` — **не колонка `Place`**, поэтому его нельзя прокинуть как остальные. Нужна отдельная ветка-композит (пропуск, который поймал Next-инженер, — иначе фильтр молча ничего не отфильтрует):

```ts
// в buildApprovedPlacesWhere
...(filter?.canWork
  ? { hasWifi: true, hasAirCon: true, hasCafeSeating: true }
  : {}),
```

### Бонус: задействовать существующий `canLeaveChild` (ноль стоимости)

Родитель-удалёнщик: половина ценности работы — «могу я реально писать код или каждые 5 минут бегу за ребёнком». Поле `canLeaveChild` **уже есть в базе**. Показывать его **видимым сигналом** на карточках, прошедших «Можно поработать». Стоит ноль, а закрывает реальную боль.

## Раздел 3. Живой индикатор статуса в карточке

### Состояния (финал): 5 значений enum, но 2 визуальных тона

Здесь критики сошлись против черновика: три градации (`open`/`openLong`/`closingSoon`) — перебор для calm-first. **`openLong` не отдельное состояние, а форматирование текста внутри `open`.**

```ts
type OpenStatus =
  | { kind: "open"; closesInMin: number }        // "Сейчас открыто" / "Открыто ещё ~N ч"
  | { kind: "closingSoon"; closesInMin: number } // "Скоро закрытие"
  | { kind: "closed"; opensTodayAt?: string }    // "Откроется в 15:00" / "Сейчас закрыто"
  | { kind: "dayOff" }                           // "Сегодня выходной"
  | { kind: "unknown" };                         // → бейдж НЕ рисуем
```

**Визуально — два тона:**
- **Спокойный (sage/нейтральный):** `open` — и «Сейчас открыто», и «Открыто ещё ~2 ч» это **один цвет**, разницу несёт текст.
- **Тёплый беж (НЕ красный, НЕ «!»):** `closingSoon`.
- `closed`/`dayOff` — серый-нейтральный (и в списке не показываем, см. ниже).
- `unknown` → **молчим** (образцовое calm-first решение из черновика, все подтвердили: лучше молчать, чем врать).

### Пороги (вынести в именованные константы в ОДНОМ файле)

```ts
// src/lib/schedule/open-status.ts — единственный источник этих чисел
const CLOSING_SOON_MIN = 90;   // ≤90 мин до закрытия → "Скоро закрытие"
const OPEN_LONG_MIN    = 120;  // ≥120 мин → показываем "ещё ~N ч" (запас: дорога + посидеть)
```

Пороги и любое «окно вечера» держим здесь с комментарием-допущением — после первой проверки реальных часов правка будет однострочной (все четыре критика настояли на «конфиг в одном месте»).

### Округление — ВНИЗ (`Math.floor`), не вверх

Черновик округлял `Math.round`: 119 мин → «~2 ч». **Все критики забраковали:** это обещание, которое родитель проверит машиной с ребёнком в +33°C, и если приедет к закрытию — это худший provenance-провал для продукта, чья ценность в доверии. **`Math.floor`:** 119 мин → «~1 ч». Недооценить безопаснее, чем переоценить. Это и есть calm-first: спокойная правда вместо оптимизма.

Число «ещё ~N ч» показываем только при `closesInMin >= 120`; между 90 и 120 — просто «Сейчас открыто» без числа.

### Точные русские формулировки (сверено с тоном `ru.ts`: «вы», нейтрально, без «!»)

```ts
// ru.ts → новый раздел openStatus
openStatus: {
  open:        "Сейчас открыто",
  openFor:     (h: number) => `Открыто ещё около ${h} ч`,   // floor, только при ≥120 мин
  closingSoon: "Скоро закрытие",
  opensAt:     (t: string) => `Откроется в ${t}`,           // утро рабочего дня
  closedToday: "Сегодня уже закрыто",                        // интервалов на сегодня больше нет
  dayOff:      "Сегодня выходной",
  // unknown → ничего не рендерим
},
```

**Критично важная поправка (родитель + UX):** черновик на месте в 10 утра (работает 15:00–20:00) показывал голое «Сейчас закрыто». Родитель читает это как «не сегодня» — дезинформация. **Правильно: «Откроется в 15:00».** Поле `opensTodayAt` в enum — это не отдельная фича, а обязательная часть корректности. Голое «Сегодня уже закрыто» — только когда интервалов на сегодня действительно больше не будет.

### Где показывать: детали — да; список — да, но только позитив

> **Здесь я меняю решение черновика.** Черновик прятал бейдж только на страницу деталей ради «кешируемости главной». **Next-инженер перепроверил и показал, что это защита несуществующего свойства:** главная — это `/[lang]/[city]` со `searchParams` фильтров и запросом в БД → в App Router она **и так динамическая**, статически не кешируется (иначе фильтры бы не работали). Бейдж не делает её «менее кешируемой».
>
> А ценность «куда успею сейчас» — это решение на уровне **списка** (родитель сканирует глазами), не деталей. Прятать бейдж на детали = заставлять открывать каждое место по очереди = анти-calm.
>
> **Решение: живой индикатор в карточках списка — уже в MVP.** Риск «половина карточек серые» снимается правилом: **в списке рисуем только позитив** (`open` / «ещё ~N ч» / `closingSoon`), а `closed`/`dayOff`/`unknown` → бейджа нет. Тогда список — не «половина серого», а «часть карточек с мягкой зелёной пометкой».
>
> **Цена, которую надо принять честно** (замечание data-архитектора): `getApprovedPlaces` сейчас **не тянет `schedules`** для списка (я проверил). Значит для бейджа в списке надо добавить `schedules: { orderBy: [...] }` в выборку списка — это +данные к каждому запросу главной. На десятках мест — незначимо. Но это реальная работа, не «бесплатно».

## Раздел 4. Техническое решение: динамика «сейчас» в Next SSR

### 4.1. Где считать — сервер на момент запроса, клиент не пересчитывает (в MVP)

Расчёт статуса — на **сервере при рендере**, по `Date.now()` на момент запроса, конвертируя в TZ города. Клиент в MVP **ничего не пересчитывает** — просто гидратирует тот же HTML.

### 4.2. `revalidate = 300` — УБРАТЬ из плана (черновик был неправ, и это важно)

> Черновик предлагал `export const revalidate = 300` «чтобы статус жил максимум 5 минут». **Next-инженер поймал здесь фактическую ошибку, и он прав:** `revalidate = N` — это ISR. Страница рендерится **один раз**, `Date.now()` **вмораживается в HTML**, и отдаётся из кеша всем до следующей фоновой перегенерации. При низком трафике (а у вас на старте он низкий) **триггера ревалидации нет — никто не заходит** → «Открыто ещё ~2 ч», отрендеренное в 14:00, спокойно доживёт до вечера. То есть `revalidate` **создаёт** ровно тот баг «замёрзшее сейчас», ради борьбы с которым введён.
>
> **Что делать: ничего не добавлять.** Я перепроверил — страницы обращаются к БД и используют динамические сегменты/`searchParams`, поэтому в App Router они **уже динамические по умолчанию** (нет `force-static`, нет закешированного `generateStaticParams`). Серверный `Date.now()` на момент запроса **уже корректен без единой строки конфигурации.** Если когда-нибудь появится агрессивное кеширование — тогда точечно `export const dynamic = "force-dynamic"` на конкретной странице. В MVP — не нужно даже этого.

### 4.3. Таймзона города — `Intl.DateTimeFormat`, без внешних библиотек ✅ (все согласны)

Единственный источник — `city.timezone` (уже в модели, «Asia/Bangkok»). Node несёт ICU, TZ-библиотеки не нужны.

```ts
// server: минуты дня + день недели в TZ города
function nowInCity(tz: string): { nowMin: number; day: DayOfWeek } {
  const parts = new Intl.DateTimeFormat("en-GB", {
    timeZone: tz, hour: "2-digit", minute: "2-digit",
    weekday: "short", hour12: false,
  }).formatToParts(new Date());
  // ... собрать hour, minute; weekday "Mon".."Sun" → enum по ЯВНОЙ таблице
}
```

**Тонкость (два критика):** маппинг `weekday` → enum `DayOfWeek` делать **явной таблицей** `{ Mon: "MON", ... }`, а НЕ `.slice().toUpperCase()` — последнее хрупко на смене локали.

### 4.4. Hydration mismatch — риск нулевой (архитектура закрывает)

Сервер отрендерил, клиент гидратирует **тот же HTML** (в MVP клиент не пересчитывает) → mismatch **невозможен**. Дрейф текста на долго открытой вкладке — принимаем как безвредный. Клиентское пообновление раз в минуту (post-MVP) делать в `useEffect` (после гидратации), первый клиентский render === серверный → mismatch тоже нет.

### 4.5. Краевые случаи (сводно) — логика черновика подтверждена всеми

```ts
function computeOpenStatus(daySchedules, nowMin, today): OpenStatus {
  const rows = daySchedules.filter(s => s.day === today);
  if (rows.length === 0) return { kind: "unknown" };
  if (rows.every(s => s.isClosed)) return { kind: "dayOff" };

  for (const s of rows.filter(r => !r.isClosed)) {
    const open = toMin(s.openTime);
    let close = toMin(s.closeTime);
    if (close <= open) close += 24 * 60;                 // закрытие после полуночи
    const nowAdj = nowMin < open ? nowMin + 24 * 60 : nowMin;
    if (nowAdj >= open && nowAdj < close) {
      const left = close - nowAdj;
      if (left <= CLOSING_SOON_MIN) return { kind: "closingSoon", closesInMin: left };
      return { kind: "open", closesInMin: left };        // "ещё ~N ч" при left>=120, иначе просто "открыто"
    }
  }
  // рабочий день, сейчас закрыто: до открытия → opensTodayAt; после — closedToday
  const next = nextOpeningToday(rows, nowMin);
  return { kind: "closed", opensTodayAt: next };
}
```

- **Полночь / закрытие после 00:00** → `close += 1440` + `nowAdj`. ✅
- **Выходной** (`isClosed`) → `dayOff`. ✅
- **Утро до открытия** → `closed` c `opensTodayAt` → «Откроется в 15:00» (не голое «закрыто»). ✅ (обязательная правка)
- **Между интервалами (обед)** → цикл по строкам дня → `closed`. ✅
- **Нет расписания** → `unknown` → молчим. ✅
- **Дубли `day`** → фикс `key` + `orderBy`. ✅

8. **Спокойный empty-state** для «Можно поработать» — «Пока таких мест немного, мы наполняем» (при узком наполнении фильтр легко даст 0–2 результата; без этого демо покажет пусто — важно именно для показа).
