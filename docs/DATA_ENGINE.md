# Движок данных: импорт мест из Google Places API

Автоматизируем «скелет» места — то, что Google знает достоверно: название,
адрес, координаты, часы работы, телефон, сайт. **Уникальная ценность гида
остаётся ручной** — описание в нашем тоне, tri-state признаки (кондиционер,
Wi-Fi, розетки…), советы «Полезно знать», цены и Thai-price, фото. Формула:
AI ≈ 70 % скелета, человек ≈ 30 % ценности — и эти 30 % и есть продукт.

## Конвейер (ничего не публикуется без человека)

```
источник (Google Places) → черновик PENDING → проверка/дозаполнение → APPROVED
```

- Импортированное место создаётся со `status: PENDING` и `sourceType: IMPORT` —
  на сайт **не попадает**, пока не одобрено (публичные выборки видят только
  APPROVED).
- Дедуп: `Place.googlePlaceId` (канонический Google Place ID, `@unique`) —
  одно заведение не занесётся дважды, даже с разными названиями в запросе.
- Одобрение и дозаполнение: `/admin` (или через seed для «богатых» карточек).
  `npm run gaps` показывает, что осталось проверить.

## Как получить API-ключ (один раз, ~10 минут)

1. <https://console.cloud.google.com> → создать проект (например,
   `pattaya-kids-guide`).
2. APIs & Services → Library → включить **Places API (New)** (именно New).
3. APIs & Services → Credentials → Create credentials → **API key**.
4. Сразу ограничить ключ: Edit key → API restrictions → Restrict key →
   отметить только Places API (New). Application restrictions → None
   (запускаем со своего компьютера).
5. Ключ положить в `.env`: `GOOGLE_PLACES_API_KEY=...` (в git не попадает).

Нужна привязанная карта (billing), но для наших объёмов это бесплатно:
у Google ежемесячный бесплатный кредит на Places API, а ручной импорт — это
единицы запросов за сессию (поиск ~цент, детали ~2 цента). Сотни импортов
в месяц остаются в пределах кредита; на всякий случай в Billing → Budgets
можно поставить бюджет-уведомление на пару долларов.

## Использование

```bash
# 1. Поиск: текст ИЛИ ссылка на карточку Google Maps (из чек-листа сбора)
npm run import:place -- "Skippy Land Lotus North Pattaya"
npm run import:place -- "https://www.google.com/maps/place/…"
#    → список кандидатов с Place ID

# 2. Превью скелета выбранного кандидата (ничего не пишет)
npm run import:place -- --id=ChIJ…

# 3. Запись черновика в базу (PENDING, на сайт не попадает)
npm run import:place -- --id=ChIJ… --write
#    другой город (когда появится): --city=<slug>
```

По умолчанию всё — dry-run. Запись только явным `--write` c явным `--id`:
«дубль страшнее потери» действует и здесь. Ссылка на карточку Maps удобнее
текста: из неё берутся и название, и координаты для locationBias (поиск не
уедет в другой город). После записи скрипт печатает чек-лист ручного
дозаполнения.

Что скрипт **не** делает намеренно: не выдумывает описание, не мапит типы
Google в наши категории (только подсказка в консоли), не ставит tri-state
признаки (они остаются `null` = «уточняется»), не скачивает фото Google
(нельзя — см. ниже).

## Медиа-права (почему у фото появились source/rightsNote)

Авторское право на фото — у того, кто снял. Для семейного бренда это
репутационно критично, поэтому фиксируем провенанс каждого фото:

- `PlacePhoto.source`: `OWN` (снято нами) / `PLACE_OWNER` (прислано местом
  с его согласия) / `CONTRIBUTOR` (прислал родитель с согласием на
  публикацию); `null` — провенанс не зафиксирован (старые записи).
- `PlacePhoto.rightsNote` / `Place.imageRightsNote`: одна строка-основание —
  «фото Вероники (визит 2026-07-07)», «согласие владельца в LINE, дата».

Можно: свои фото; фото от самих мест с их «ок»; фото контрибьюторов с
согласием. Нельзя: чужие фото из Google/Instagram перезаливать к себе (фото
из Places API разрешено только показывать через их API с атрибуцией — не
сохранять в свою галерею); узнаваемые чужие дети без согласия — никогда.

## Порядок деплоя этого среза (важно!)

Схема получила новые колонки/enum. Прод не сломается, только если соблюсти
порядок:

1. **Сначала** миграция базы (добавляет nullable-колонки — данные целы):
   `npx prisma db push` (из папки проекта, `.env` смотрит в прод-базу).
2. **Потом** merge PR (Vercel задеплоит код, знающий новые колонки).

Наоборот нельзя: задеплоенный код начнёт SELECT'ить колонки, которых ещё нет
в базе, и страницы каталога упадут.
